Webpack provides an \href{https://webpack.js.org/concepts/module-resolution/}{\texttt{ advanced mechanism to resolve files}}.

The {\ttfamily sass-\/loader} uses Sass\textquotesingle{}s custom importer feature to pass all queries to the Webpack resolving engine. Thus you can import your Sass modules from {\ttfamily node\+\_\+modules}.


\begin{DoxyCode}{0}
\DoxyCodeLine{@import "{}bootstrap"{};}

\end{DoxyCode}


Using {\ttfamily $\sim$} is deprecated and can be removed from your code ({\bfseries{we recommend it}}), but we still support it for historical reasons. Why can you remove it? The loader will first try to resolve {\ttfamily @import} as a relative path. If it cannot be resolved, then the loader will try to resolve {\ttfamily @import} inside \href{https://webpack.js.org/configuration/resolve/\#resolvemodules}{\texttt{ {\ttfamily node\+\_\+modules}}}.

Prepending module paths with a {\ttfamily $\sim$} tells webpack to search through \href{https://webpack.js.org/configuration/resolve/\#resolvemodules}{\texttt{ {\ttfamily node\+\_\+modules}}}.


\begin{DoxyCode}{0}
\DoxyCodeLine{@import "{}\string~bootstrap"{};}

\end{DoxyCode}


It\textquotesingle{}s important to prepend it with only {\ttfamily $\sim$}, because {\ttfamily $\sim$/} resolves to the home directory. Webpack needs to distinguish between {\ttfamily bootstrap} and {\ttfamily $\sim$bootstrap} because CSS and Sass files have no special syntax for importing relative files. Writing {\ttfamily @import \char`\"{}style.\+scss\char`\"{}} is the same as {\ttfamily @import \char`\"{}./style.\+scss\char`\"{};}\hypertarget{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules_sass_loader__r_e_a_d_m_e_autotoc_md12529}{}\doxysubsubsection{Problems with $<$tt$>$url(...)$<$/tt$>$}\label{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules_sass_loader__r_e_a_d_m_e_autotoc_md12529}
Since Sass implementations don\textquotesingle{}t provide \href{https://github.com/sass/libsass/issues/532}{\texttt{ url rewriting}}, all linked assets must be relative to the output.


\begin{DoxyItemize}
\item If you pass the generated CSS on to the {\ttfamily css-\/loader}, all urls must be relative to the entry-\/file (e.\+g. {\ttfamily main.\+scss}).
\item If you\textquotesingle{}re just generating CSS without passing it to the {\ttfamily css-\/loader}, it must be relative to your web root.
\end{DoxyItemize}

You will be disrupted by this first issue. It is natural to expect relative references to be resolved against the {\ttfamily .sass}/{\ttfamily .scss} file in which they are specified (like in regular {\ttfamily .css} files).

Thankfully there are a two solutions to this problem\+:


\begin{DoxyItemize}
\item Add the missing url rewriting using the \href{https://github.com/bholloway/resolve-url-loader}{\texttt{ resolve-\/url-\/loader}}. Place it before {\ttfamily sass-\/loader} in the loader chain.
\item Library authors usually provide a variable to modify the asset path. \href{https://github.com/twbs/bootstrap-sass}{\texttt{ bootstrap-\/sass}} for example has an {\ttfamily \$icon-\/font-\/path}.
\end{DoxyItemize}\hypertarget{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules_sass_loader__r_e_a_d_m_e_autotoc_md12530}{}\doxysubsection{Options}\label{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules_sass_loader__r_e_a_d_m_e_autotoc_md12530}
