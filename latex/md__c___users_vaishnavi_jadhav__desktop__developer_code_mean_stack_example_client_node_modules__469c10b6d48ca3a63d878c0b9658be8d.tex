Schematics are generators that transform an existing filesystem. They can create files, refactor existing files, or move files around.

What distinguishes Schematics from other generators, such as Yeoman or Yarn Create, is that schematics are purely descriptive; no changes are applied to the actual filesystem until everything is ready to be committed. There is no side effect, by design, in Schematics.\hypertarget{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules__469c10b6d48ca3a63d878c0b9658be8d_autotoc_md407}{}\doxysection{Glossary}\label{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules__469c10b6d48ca3a63d878c0b9658be8d_autotoc_md407}

Schematics is a library, and does not work by itself. A \href{https://github.com/angular/angular-cli/blob/master/packages/angular_devkit/schematics_cli/bin/schematics.ts}{\texttt{ reference CLI}} is available on this repository, and is published on NPM at \href{https://www.npmjs.com/package/@angular-devkit/schematics-cli}{\texttt{ @angular-\/devkit/schematics-\/cli}}. This document explains the library usage and the tooling API, but does not go into the tool implementation itself.

The tooling is responsible for the following tasks\+:


\begin{DoxyEnumerate}
\item Create the Schematic Engine, and pass in a Collection and Schematic loader.
\end{DoxyEnumerate}
\begin{DoxyEnumerate}
\item Understand and respect the Schematics metadata and dependencies between collections. Schematics can refer to dependencies, and it\textquotesingle{}s the responsibility of the tool to honor those dependencies. The reference CLI uses NPM packages for its collections.
\end{DoxyEnumerate}
\begin{DoxyEnumerate}
\item Create the Options object. Options can be anything, but the schematics can specify a JSON Schema that should be respected. The reference CLI, for example, parses the arguments as a JSON object and validates it with the Schema specified by the collection.
\end{DoxyEnumerate}
\begin{DoxyEnumerate}
\item Schematics provides some JSON Schema formats for validation that tooling should add. These validate paths, html selectors and app names. Please check the reference CLI for how these can be added.
\end{DoxyEnumerate}
\begin{DoxyEnumerate}
\item Call the schematics with the original Tree. The tree should represent the initial state of the filesystem. The reference CLI uses the current directory for this.
\end{DoxyEnumerate}
\begin{DoxyEnumerate}
\item Create a Sink and commit the result of the schematics to the Sink. Many sinks are provided by the library; File\+System\+Sink and Dry\+Run\+Sink are examples.
\end{DoxyEnumerate}
\begin{DoxyEnumerate}
\item Output any logs propagated by the library, including debugging information.
\end{DoxyEnumerate}

A {\ttfamily Source} is a generator of a {\ttfamily Tree}; it creates an entirely new root tree from nothing. A {\ttfamily Rule} is a transformation from one {\ttfamily Tree} to another. A {\ttfamily Schematic} (at the root) is a {\ttfamily Rule} that is normally applied on the filesystem.\hypertarget{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules__469c10b6d48ca3a63d878c0b9658be8d_autotoc_md414}{}\doxysubsection{Operators}\label{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules__469c10b6d48ca3a63d878c0b9658be8d_autotoc_md414}
{\ttfamily File\+Operator}s apply changes to a single {\ttfamily File\+Entry} and return a new {\ttfamily File\+Entry}. The result follows these rules\+:


\begin{DoxyEnumerate}
\item If the {\ttfamily File\+Entry} returned is null, a {\ttfamily Delete\+Action} will be added to the action list.
\end{DoxyEnumerate}
\begin{DoxyEnumerate}
\item If the path changed, a {\ttfamily Rename\+Action} will be added to the action list.
\end{DoxyEnumerate}
\begin{DoxyEnumerate}
\item If the content changed, an {\ttfamily Overwrite\+Action} will be added to the action list.
\end{DoxyEnumerate}
