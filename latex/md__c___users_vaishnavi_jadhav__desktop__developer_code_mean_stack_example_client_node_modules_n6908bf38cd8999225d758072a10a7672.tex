{\ttfamily npm} bundles its own, internal, copy of {\ttfamily node-\/gyp}. This internal copy is independent of any globally installed copy of node-\/gyp that may have been installed via {\ttfamily npm install -\/g node-\/gyp}.

Generally, npm\textquotesingle{}s library files are installed inside your global \char`\"{}node\+\_\+modules\char`\"{}, where npm is installed (run {\ttfamily npm prefix} and add {\ttfamily lib/node\+\_\+modules}, or just {\ttfamily node\+\_\+modules} for Windows). There are some exceptions to this. Inside this global {\ttfamily node\+\_\+modules/} there will be an {\ttfamily npm/} directory and inside this you\textquotesingle{}ll find a {\ttfamily node\+\_\+modules/node-\/gyp/} directory. So it may look something like {\ttfamily /usr/local/lib/node\+\_\+modules/npm/node\+\_\+modules/node-\/gyp/}. This is the version of node-\/gyp that ships with npm.

When you install a {\itshape new} version of node-\/gyp outside of npm, it\textquotesingle{}ll go into your global node\+\_\+modules, but not under the {\ttfamily npm/node\+\_\+modules}. So that may look like {\ttfamily /usr/local/lib/node\+\_\+modules/node-\/gyp/}. It\textquotesingle{}ll have the {\ttfamily node-\/gyp} executable linked into your {\ttfamily PATH} so running {\ttfamily node-\/gyp} will use this version.

The catch is that npm won\textquotesingle{}t use this version unless you tell it to, it\textquotesingle{}ll keep on using the one you have installed. You need to instruct it to by setting the {\ttfamily node\+\_\+gyp} config variable (which goes into your {\ttfamily $\sim$/.npmrc}). You do this by running the {\ttfamily npm config set} command as below. Then npm will use the command in the path you supply whenever it needs to build a native addon.

{\bfseries{Important}}\+: You also need to remember to unset this when you upgrade npm with a newer version of node-\/gyp, or you have to manually keep your globally installed node-\/gyp to date.
