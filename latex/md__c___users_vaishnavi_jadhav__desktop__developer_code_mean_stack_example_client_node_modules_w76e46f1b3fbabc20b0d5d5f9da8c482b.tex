An express-\/style development middleware for use with \href{https://webpack.js.org}{\texttt{ webpack}} bundles and allows for serving of the files emitted from webpack. This should be used for {\bfseries{development only}}.

Some of the benefits of using this middleware include\+:


\begin{DoxyItemize}
\item No files are written to disk, rather it handles files in memory
\item If files changed in watch mode, the middleware delays requests until compiling has completed.
\item Supports hot module reload (HMR).
\end{DoxyItemize}\hypertarget{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules_w76e46f1b3fbabc20b0d5d5f9da8c482b_autotoc_md15356}{}\doxysubsection{Getting Started}\label{md__c___users_vaishnavi_jadhav__desktop__developer_code_mean_stack_example_client_node_modules_w76e46f1b3fbabc20b0d5d5f9da8c482b_autotoc_md15356}

In order to develop an app using server-\/side rendering, we need access to the \href{https://github.com/webpack/docs/wiki/node.js-api\#stats}{\texttt{ {\ttfamily stats}}}, which is generated with each build.

With server-\/side rendering enabled, {\ttfamily webpack-\/dev-\/middleware} sets the {\ttfamily stats} to {\ttfamily res.\+locals.\+webpack.\+dev\+Middleware.\+context.\+stats} and the filesystem to {\ttfamily res.\+locals.\+webpack.\+dev\+Middleware.\+context.\+output\+File\+System} before invoking the next middleware, allowing a developer to render the page body and manage the response to clients.

{\itshape Note\+: Requests for bundle files will still be handled by {\ttfamily webpack-\/dev-\/middleware} and all requests will be pending until the build process is finished with server-\/side rendering enabled.}
